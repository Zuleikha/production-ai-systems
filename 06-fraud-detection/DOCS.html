Imbalance Handling

Three approaches compared: SMOTE (synthetic oversampling of minority class), random undersampling of majority class, and class_weight="balanced". SMOTE + balanced weights gave the best lift (+0.03 PR-AUC over baseline). The SMOTE step is inside the training pipeline so it only sees training data — never validation or test.

Data & Class Imbalance

The Dataset

284,807 credit card transactions from European cardholders over two days. 492 are fraud (0.17%). Features V1–V28 are PCA components — the original features were anonymised. Amount is the transaction amount (log-scaled to log_amount). Time is seconds from the first transaction (converted to hour_of_day).

Source: Kaggle — Credit Card Fraud Detection (MLG-ULB)

Why 99.83% accuracy means nothing here

A model that always predicts "legitimate" scores 99.83% accuracy. It catches zero fraud. This is why we use PR-AUC (Precision-Recall Area Under Curve) as the primary metric — it focuses entirely on the minority class and is not flattered by the majority.

Feature Engineering

Raw transaction fields alone are weak signal. Context from the card's history is what separates fraud from a legitimate unusual purchase.

Time Features

  • hour_of_day — fraud spikes at unusual hours (2–4am). A single field but a strong signal.
  • day_of_week — weekend patterns differ from weekdays for both legitimate and fraudulent behaviour.

Velocity Features

Computed from the card's recent transaction history. A fraudster who steals a card often does multiple small test transactions before a large one — transactions_last_1h = 5 + amount = €0.01 is a classic card-testing signature.

StandardScaler isn't strictly needed for XGBoost (tree-based models are scale-invariant) but keeps the pipeline consistent if you swap algorithms. handle_unknown='ignore' on the encoder means unseen merchant categories at inference time don't crash the API.

Production note: feature stores

In a real system, velocity features live in a feature store (Redis, Feast, Tecton). At inference time you look up card_id and get pre-computed values in <5ms. In this project they're computed offline for training — this is an acknowledged limitation.

Optuna Hyperparameter Search

Optuna runs 150 trials, each with a different combination of hyperparameters sampled from defined ranges. Pruning stops unpromising trials early. Every trial is logged to MLflow.

def objective(trial):
    params = {
        "learning_rate":    trial.suggest_float("learning_rate", 0.01, 0.3, log=True),
        "max_depth":        trial.suggest_int("max_depth", 3, 10),
        "n_estimators":     trial.suggest_int("n_estimators", 100, 500),
        "subsample":        trial.suggest_float("subsample", 0.6, 1.0),
        "colsample_bytree": trial.suggest_float("colsample_bytree", 0.6, 1.0),
        "gamma":            trial.suggest_float("gamma", 0, 5),
        "min_child_weight": trial.suggest_int("min_child_weight", 1, 10),
    }
    model = XGBClassifier(**params, eval_metric="aucpr", ...)
    # 5-fold cross-validated PR-AUC
    return cross_val_score(model, X_train, y_train,
                           scoring="average_precision", cv=5).mean()

Model Evaluation

Why accuracy is the wrong metric

A model predicting "legitimate" for every transaction scores 99.83% accuracy and catches zero fraud. The relevant metrics for imbalanced classification are:

MetricFormulaWhat it measures
PrecisionTP / (TP + FP)Of transactions flagged as fraud, how many actually were? Low = too many false alarms.
RecallTP / (TP + FN)Of actual fraud cases, how many were caught? Low = fraud slips through.
PR-AUCArea under P-R curveOverall model quality on the minority class, independent of threshold. Primary metric here.
ROC-AUCArea under ROC curveGood for model comparison. Less informative than PR-AUC with heavy imbalance.
F1Harmonic mean of P & RUseful single number but hides the precision-recall tradeoff.

Threshold Optimisation

Why 0.5 is almost never right in production

sklearn's default threshold of 0.5 assumes equal cost for false positives and false negatives. In fraud detection, a missed fraud (FN) costs roughly 10–30× more than a false block (FP). With extreme class imbalance, predicted probabilities cluster near 0 — a threshold of 0.5 will flag almost nothing.

Business Cost Matrix Sweep

fn_cost = 150   # fraud gets through — financial loss
fp_cost = 5     # customer wrongly blocked — support cost + churn risk

costs = []
thresholds = np.linspace(0.01, 0.99, 100)

for t in thresholds:
    y_pred = (y_prob >= t).astype(int)
    fp = ((y_pred == 1) & (y_test == 0)).sum()
    fn = ((y_pred == 0) & (y_test == 1)).sum()
    costs.append(fp * fp_cost + fn * fn_cost)

optimal_threshold = thresholds[np.argmin(costs)]
# → 0.28

fn_cost / fp_cost = 30 means each missed fraud is 30× more expensive than a false alarm. This pushes the optimal threshold down from 0.5 to 0.28.

Threshold is a business decision, not a technical one

The threshold is stored separately from the model artefact. Changing the threshold means updating a config value and restarting the server — not retraining the model. In production, threshold changes typically go through A/B tests or champion/challenger frameworks before being applied globally.

Q&A

Why PR-AUC instead of ROC-AUC as the primary metric?

ROC-AUC measures the tradeoff between true positive rate and false positive rate at all thresholds. With 99.83% legitimate transactions, it's very easy to get a high ROC-AUC — the model just needs to avoid predicting fraud for the vast majority of legitimate cases, which is trivial. PR-AUC specifically measures performance on the minority (fraud) class by looking at the precision-recall tradeoff. A model that catches no fraud at all can score well on ROC-AUC but will have near-zero PR-AUC. PR-AUC is a much harder and more honest measure here.

Why does SMOTE go inside the training pipeline, not outside it?

If SMOTE runs before the train/test split, synthetic examples generated from training data leak information into the test set — they're derived from real training examples, so the model has effectively "seen" part of the test set. Putting SMOTE inside the imblearn.Pipeline ensures it only sees training folds during cross-validation, and is not applied at all to the validation or test set. This is the correct way to prevent data leakage with oversampling.

What is SHAP and why does fraud detection specifically need it?

SHAP (SHapley Additive exPlanations) decomposes each prediction into per-feature contributions — how much did V14 push the fraud probability up? How much did the hour_of_day push it down? For XGBoost (a black-box model), this provides post-hoc interpretability.

Fraud detection needs this for two reasons:

  • Regulatory compliance: financial institutions are often required to explain why a transaction was declined. "The model said so" is not acceptable.
  • Operational trust: fraud analysts reviewing flagged transactions need to see the reasoning to decide whether to act. Without explanation, they can't calibrate trust in the system.
What is Optuna and how is it different from a grid search?

Grid search exhaustively evaluates every combination in a predefined grid — if you have 5 values for 7 parameters, that's 78,125 evaluations. Optuna uses Bayesian optimisation: after each trial, it updates a probabilistic model of which parameter regions produce good results, and samples the next trial from promising regions. This means:

  • It finds good hyperparameters in far fewer trials than grid search
  • It can search continuous ranges, not just fixed grids
  • It uses pruning to stop bad trials early, saving time

150 Optuna trials covers the search space far more intelligently than a grid search of comparable size.

What do the V1–V28 features actually represent?

V1–V28 are PCA (Principal Component Analysis) components of the original transaction features. The dataset provider (ULB) applied PCA before releasing it publicly to protect cardholder privacy — the original features (merchant name, card number, location, etc.) are not disclosed. This means we can train a working model but we can't interpret what V14 or V3 means in business terms. In a real deployment you'd have access to the original features, and SHAP interpretations would make direct business sense.

Why is log_amount used instead of raw amount?

Transaction amounts are highly skewed — most transactions are small (€5–€200) but occasional large ones (€5,000+) would dominate the feature space. Log-transforming the amount compresses the scale so the model treats the difference between €5 and €50 similarly to the difference between €500 and €5,000 — both are an order of magnitude. This is a standard preprocessing step for monetary values in financial ML.

How does the threshold value get into the API?

The threshold is hardcoded as a constant in app/main.py (THRESHOLD = 0.28). It's separate from the model pickle — changing the threshold doesn't require retraining, just redeploying the API with a new value. In a more mature system you'd read it from a config file or environment variable so it can be changed without a code change. The model artefact (outputs/models/best_xgb.pkl) encodes the learned weights — the threshold encodes the business decision about acceptable precision/recall tradeoff.

Why is the dataset 284,807 rows but only 492 fraud cases?

This reflects real-world credit card fraud rates. In mature payment systems with fraud controls already in place, the fraud rate is typically 0.1%–0.5%. The ULB dataset was collected from European cardholders in September 2013 — 492 fraudulent transactions out of 284,807 total gives a fraud rate of 0.172%. The extreme imbalance (575:1 legitimate-to-fraud ratio) is not a dataset quirk; it's the actual distribution a production fraud model faces.

What would a production version of this system add?

This project is a complete end-to-end ML pipeline but a production deployment would also include:

  • Real-time feature store (Redis/Feast) for velocity features at <5ms latency
  • Streaming pipeline (Kafka + Flink) for sub-second decision-making
  • Model retraining pipeline triggered by data drift detection
  • Champion/challenger framework for A/B testing threshold and model changes
  • Graph neural network layer — GNNs catch ring fraud patterns that tabular models miss
  • Feedback loop — confirmed fraud labels from chargeback outcomes fed back into retraining
How much does the Render deployment cost?

The API is deployed on Render's free tier — $0/month. Limitations: 512MB RAM, instance spins down after 15 minutes of inactivity (first request after sleep takes ~30 seconds to wake). The XGBoost model loads from a pickle file at startup — no GPU needed, RAM usage is well within the 512MB limit. Limitations & Honest Notes

Velocity features are computed offline. In this project, velocity features (transactions in the last hour, etc.) are computed from the dataset at training time. In a real system, these would be served from a feature store with pre-aggregated values per card — you don't have the card history available at inference time in a live payment system.
V1–V28 are PCA-transformed. The original features are anonymised. This means SHAP explanations reference abstract components, not interpretable business concepts like merchant category or location. Real fraud systems have access to the original features.
No concept drift handling. Fraud patterns change as fraudsters adapt. This model was trained on 2013 data and has no retraining trigger. A production system needs monitoring for data drift and scheduled or trigger-based retraining.
No streaming. The API scores one transaction at a time via REST call. Real-time fraud detection at scale uses a streaming pipeline (Kafka + Flink) with sub-100ms decision requirements — REST is too slow for payment authorisation windows.
Render free tier spins down. The first request after 15 minutes of inactivity takes ~30 seconds. Not suitable for a real payment API — included here for portfolio demonstration only.
