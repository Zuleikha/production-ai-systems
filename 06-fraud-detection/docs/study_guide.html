📊 Exploratory Data Analysis Notebook 01

▼

The dataset

284,807 transactions. Only 492 are fraud — that is 0.17%. This extreme imbalance is not a data problem. It is reality.

⚠️ Why accuracy is useless here

A model that predicts "legit" for every single transaction scores 99.83% accuracy and catches zero fraud. Accuracy is a broken metric when classes are imbalanced.

EDA checklist — always do this before touching a model

  • Check for missing values — are any features always null for fraud?
  • Plot amount distribution — fraud often clusters at unusual amounts
  • Look at time patterns — fraud spikes at certain hours
  • Check correlation with label — which raw features separate fraud from legit?

Quick-check questions

Why can we not use accuracy to measure our fraud model? +
Because the data is so imbalanced. A model that predicts "legit" for every transaction gets 99.83% accuracy while catching zero fraud. Accuracy rewards predicting the majority class, so it hides terrible performance on the minority class (fraud).
What is class imbalance and why does it matter? +
Class imbalance is when one outcome is far rarer than the other. Here fraud is 0.17% of data. It matters because most algorithms optimise for overall accuracy, so they learn to ignore the rare class entirely unless we correct for it.

🔧 Feature Engineering Notebook 01 / 03

▼

A single transaction row tells you: amount, merchant, card type, time. That is not enough signal. A €500 electronics purchase at 2am is suspicious — but only if the card normally makes €30 supermarket purchases at noon. Context comes from history.

1. Time features

  • hour_of_day (0–23) — fraud spikes at unusual hours
  • day_of_week (0–6) — weekend patterns differ

2. Velocity features

FeatureWindowWhat it captures
transactions_last_1h1 hourBurst activity — card testing
transactions_last_24h24 hoursDaily usage anomaly
amount_mean_last_24h24 hoursNormal spend level
amount_std_last_24h24 hoursSpend variability
🔍 Classic fraud signature

transactions_last_1h = 5 + amount = €0.01 is card-testing. A fraudster does multiple tiny transactions to check if a stolen card is live before making one big purchase.

3. Categorical features

  • merchant_categoryGroups like gambling, electronics, travel that carry different fraud risk profiles. — one-hot encoded
  • card_type (credit / debit / prepaid) — one-hot encoded

The preprocessing pipeline

ColumnTransformer([
    ("num", StandardScaler(),   numeric_features),
    ("cat", OneHotEncoder(...), categorical_features),
])

StandardScalerTransforms each numeric column to have mean=0 and standard deviation=1. Required for distance-based models like logistic regression. is essential for logistic regression and SVM. RandomForest doesn't need it — but it doesn't hurt and keeps the pipeline consistent if you swap algorithms later.

Quick-check questions

What are velocity features and why are they important? +
Velocity features count or summarise a card's recent activity (e.g. number of transactions in the last hour). They are important because a single transaction looks normal in isolation — but 5 transactions in 10 minutes is a red flag for card-testing fraud.
Why must SMOTE be applied inside the training fold only? +
If you apply SMOTE before the train/test split, synthetic fraud examples from the training set leak into the test set. Your evaluation metrics will look better than they really are — a false sense of confidence.

📌 stratify=y matters

Always use train_test_split(..., stratify=y). Without it a random split might put all 492 fraud cases in training and zero in the test set — then you cannot evaluate at all.

Quick-check questions

Why is Random Forest a good starting point rather than XGBoost? +
Random Forest needs minimal tuning and produces useful results quickly. It is better to get a working pipeline end-to-end first, then swap in XGBoost once you have a solid baseline to compare against.
What does class_weight="balanced" do? +
It tells the algorithm to penalise misclassifying the minority class (fraud) proportionally more than the majority class (legit). If fraud is 1/588 of data, each fraud mistake counts 588× more in the loss function.

⚖️ Handling Class Imbalance Notebook 03

▼

Three strategies

StrategyWhat it doesTrade-off
Class weightsPenalises misclassifying minority class moreFast, no new samples
UndersamplingRandomly drop majority class rowsLoses information
SMOTESynthetic Minority Oversampling TEchnique. Creates new synthetic fraud examples by interpolating between existing ones.Synthesises new minority class examplesAdds information, risk of overfitting
🛡️ The golden rule of SMOTE

Apply SMOTE inside the training pipeline, on the training fold only. Never before the train/test split. If you do it before the split, synthetic examples contaminate the test set and your evaluation metrics are optimistically wrong.

Why we use both SMOTE and class_weight

With 0.17% fraud rate the imbalance is extreme. Using both is "belt-and-braces" — in practice one or the other is usually enough, but here we use both because the minority class is so rare.

Quick-check questions

What is the difference between SMOTE and undersampling? +
SMOTE creates new synthetic examples of the minority class (fraud) by interpolating between existing ones — you end up with more data. Undersampling randomly removes examples from the majority class (legit) — you end up with less data and potentially lose real information.

🚀 XGBoost Tuning Notebook 04

▼

XGBoosteXtreme Gradient Boosting — builds trees sequentially, each one correcting the errors of the previous. State-of-the-art for tabular data competitions. is the step up from Random Forest. It builds trees sequentially rather than in parallel — each tree learns to fix the mistakes of the previous ones.

Key XGBoost parameters

ParameterWhat it controlsStarting value

📌 Use early stopping

Set early_stopping_rounds=50. XGBoost will stop adding trees when the validation score stops improving — prevents overfitting and saves time.

Quick-check questions

How does XGBoost differ from Random Forest? +
Random Forest builds many trees in parallel — each independently. XGBoost builds trees sequentially — each new tree focuses on the residual errors left by all previous trees. This makes XGBoost more accurate but also more prone to overfitting if not tuned carefully.

Quick-check questions

What is the business cost of a False Negative vs a False Positive? +
False Negative (FN): fraud slips through → direct financial loss (the bank/merchant absorbs the fraudulent charge). False Positive (FP): a legitimate customer is blocked → frustration, support call costs, and risk of losing that customer.
Why is PR-AUC better than ROC-AUC for fraud detection? +
ROC-AUC includes True Negatives in its calculation — and we have 284,000 of them. A high TN count inflates ROC-AUC and hides poor performance on fraud. PR-AUC only looks at precision and recall, which are both about the minority class, so it gives a more honest picture.

🔍 SHAP Explainability Notebook 05

▼

SHAPSHapley Additive exPlanations — from game theory. Assigns each feature a contribution score for a specific prediction. Positive = pushed toward fraud. Negative = pushed toward legit. (SHapley Additive exPlanations) answers: why did the model predict this transaction is fraud?

What SHAP tells you

  • Global importance: which features matter most across all predictions (beeswarm plot, bar chart)
  • Local explanation: why the model scored this specific transaction as fraud (waterfall / force plot)
📌 What we found

Velocity features — especially transactions_last_1h and amount_mean_last_24h — are the top fraud signals. The engineered features matter more than the raw transaction fields. This confirms that context beats a snapshot.

Reading a SHAP waterfall chart

  • Red bars (positive SHAP value) — pushed the score toward fraud
  • Blue bars (negative SHAP value) — pushed the score toward legit
  • The sum of all SHAP values = final prediction - average prediction

Key SHAP plots

PlotWhat it shows
Bar chartMean absolute SHAP value per feature — global ranking
BeeswarmDistribution of SHAP values for each feature across all samples
WaterfallPer-prediction breakdown — which features pushed this score up/down
Dependence plotHow a single feature's SHAP value changes across its value range

Quick-check questions

What does a SHAP value of +0.4 for transactions_last_1h mean? +
It means that the value of transactions_last_1h for this particular transaction pushed the model's fraud score up by 0.4 log-odds (or probability units, depending on the model type). The feature increased the fraud probability for this transaction.
Why are SHAP values useful in production fraud detection? +
When a customer calls to dispute a blocked transaction, you can tell them exactly why — "this transaction was flagged because your card made 4 transactions in the last hour, which is unusual for your account." It makes the model's decision auditable and explainable to non-technical stakeholders.

🎚️ Threshold Tuning Notebook 06

▼

When sklearn calls model.predict(), the default cutoff is 0.5. For fraud detection this is almost always wrong. With a 0.17% fraud rate, model probabilities cluster near 0 — a threshold of 0.5 flags almost nothing.

💡 The fundamental tradeoff
  • Lower threshold → recall rises (catch more fraud) → precision falls (more false alarms)
  • Higher threshold → precision rises (fewer false alarms) → recall falls (miss more fraud)
  • There is no free lunch. Every threshold is a business decision, not a technical one.

Cost-based threshold selection

fn_cost = 150   # fraud gets through — bank absorbs loss
fp_cost = 5     # customer wrongly blocked — support + churn risk

optimal_threshold = threshold that minimises (FP × fp_cost + FN × fn_cost)

With fn_cost / fp_cost = 30, each missed fraud is 30× more expensive than a false alarm. This pushes the threshold down — you'd rather block 30 legitimate transactions than miss one fraud case.

Where the threshold lives

# config/config.yaml
prediction:
  threshold: 0.3

The threshold is a config value — not hardcoded. You can change it without retraining the model or redeploying the artefact. This separation is intentional: the model artefact captures learned patterns; the threshold captures a business decision.

Quick-check questions

If fn_cost is 5× higher than fp_cost, should the threshold be higher or lower than 0.5? +
Lower. A higher fn_cost means missing fraud is much more expensive than a false alarm. To catch more fraud (increase recall) you lower the threshold — accepting more false positives in exchange for fewer missed fraud cases.
Why is 0.5 almost always the wrong threshold for fraud detection? +
With only 0.17% fraud, the model learns that most transactions are legit, so it outputs low fraud probabilities across the board. A threshold of 0.5 will flag almost nothing because very few predictions exceed 0.5. You need a much lower threshold, e.g. 0.3 or 0.28.

💡 Load the model once at startup

Use FastAPI's lifespan context manager to load the model when the server starts. Every request then reuses the same in-memory object. If you loaded inside the endpoint function, you'd pay a ~200ms disk read on every single call.

The /health endpoint

@app.get("/health")
def health():
    return {"status": "ok"}

This exists for the load balancer. When Render (or Kubernetes) deploys your service, it polls /health continuously. If it stops returning 200, the platform stops routing traffic to that instance and may restart it. This means new deployments only receive traffic after they pass the health check.

Quick-check questions

Why must the model be loaded at startup rather than per request? +
Loading a scikit-learn model from disk takes ~200ms. At 100 requests per second that adds 20 seconds of wasted load time per second — the service would fall over. Loading once at startup and reusing the in-memory object costs near zero per request.
What does Pydantic do in the API? +
Pydantic validates every incoming request against a schema. If a field is missing, the wrong type, or out of range, it immediately returns HTTP 422 — before the request ever reaches the model. This prevents the model from receiving bad data and crashing with a cryptic error.

📡 Production Monitoring Notebook 07

▼

Once the service is running, these are the first three things a senior ML engineer adds.

1. Prediction distribution monitoring

Log every fraud_probability value. Watch the daily mean. If it shifts — say, from 0.12 to 0.05 — something changed: fraud patterns, input features, or a bug.

2. Data drift detection

Compare incoming feature distributions to the training distribution. If amount_mean_last_24h starts looking different, the model is predicting on data it was not built for. Tools: Evidently AI, WhyLabs, or a simple Kolmogorov-Smirnov test.

3. Label feedback loop

In fraud detection, labels arrive late — a transaction confirmed fraudulent today was made three weeks ago. Build a pipeline to:

  1. Collect confirmed fraud labels from the chargebacks team
  2. Join them back to your logged predictions
  3. Compute actual precision and recall over the last 30 days
  4. Alert if recall drops below a threshold (e.g. < 85%)
⚠️ Without a label feedback loop you are flying blind

The model might be degrading for weeks before anyone notices. Fraud patterns shift constantly. Without comparing predictions to confirmed outcomes, you have no way to detect that recall has silently dropped.

Quick-check questions

What is data drift and why is it dangerous for a fraud model? +
Data drift is when the statistical distribution of incoming features changes compared to what the model was trained on. Fraud patterns evolve constantly — new attack vectors, new merchant categories, seasonal patterns. A model trained six months ago may be silently making worse predictions today without anyone noticing.
Why do fraud labels arrive late, and how does this affect monitoring? +
A fraud chargeback can take weeks to be filed, investigated, and confirmed. So at the moment you make a prediction, you don't know if it was right. You can only evaluate accuracy retrospectively — which means your monitoring pipeline must join predictions to delayed labels, not evaluate in real time.

🎤 Interview Prep — Q&A All notebooks

▼

Click any question to reveal a model answer. These cover the concepts most likely to come up when presenting or defending this project.

1
Why can't you use accuracy to evaluate a fraud detection model?
+
Model answer
Because the dataset is severely imbalanced — 99.83% of transactions are legitimate. A model that predicts "legit" for every single transaction scores 99.83% accuracy while catching zero fraud. Accuracy rewards predicting the majority class and completely hides failure on the minority class. Instead we use precision, recall, F1, ROC-AUC, and PR-AUC — metrics that are sensitive to minority class performance.
2
What is SMOTE and why must it only be applied inside the training fold?
+
Model answer
SMOTE (Synthetic Minority Oversampling TEchnique) creates new synthetic fraud examples by interpolating between existing ones, so the model has more minority class data to learn from. It must be applied inside the training pipeline, after the train/test split. If you apply it before splitting, synthetic examples generated from training data leak into the test set — your evaluation metrics look better than they really are because the test set is no longer truly unseen.
3
Explain precision and recall in plain language. Which matters more for fraud?
+
Model answer
Precision asks: of all the transactions I flagged as fraud, how many actually were? Low precision means too many false alarms — legitimate customers get blocked, get frustrated, and churn.

Recall asks: of all the actual fraud cases, how many did I catch? Low recall means fraud slips through undetected — a direct financial loss.

Neither is universally more important. It depends on the cost ratio. If missing one fraud case costs £150 and blocking one legitimate customer costs £5, you accept more false alarms to catch more fraud — you optimise recall. The threshold is how you control that tradeoff.
4
Why is the default 0.5 classification threshold almost always wrong for this problem?
+
Model answer
With only 0.17% fraud, the model learns that almost every transaction is legitimate, so the fraud probabilities it outputs cluster near zero. A 0.5 threshold flags almost nothing — you'd miss most fraud. The right threshold is determined by the business cost ratio: if a missed fraud costs 30× more than a false alarm, you lower the threshold to roughly 0.3 so the model catches more fraud at the cost of more false alarms. In this project we found the cost-optimal threshold is 0.28.
5
Walk me through how you choose an optimal threshold using business costs.
+
Model answer
You define two costs: fn_cost (the average loss when fraud slips through — e.g. £150) and fp_cost (the cost of wrongly blocking a customer — e.g. £5). Then for each candidate threshold you compute: total_cost = FP × fp_cost + FN × fn_cost on the validation set. The threshold that minimises total cost is your operating point. This is purely a business decision, not a model decision — you can change it any time by editing the config file without retraining.
6
What are velocity features and why did they turn out to be the top fraud signals?
+
Model answer
Velocity features summarise a card's recent transaction history — for example, how many transactions occurred in the last hour, or the card's average spend over the last 24 hours. They matter because a single transaction looks normal in isolation. A £500 electronics purchase at 2am is suspicious only if the card normally makes £30 supermarket purchases at noon. Velocity gives the model that context. SHAP analysis confirmed that transactions_last_1h and amount_mean_last_24h were the strongest fraud signals — more important than the raw transaction fields.
7
What is SHAP and how would you use it in a production fraud system?
+
Model answer
SHAP assigns each feature a contribution score for a specific prediction — positive values push toward fraud, negative toward legit. It comes from game theory (Shapley values) and is model-agnostic. In production it has two uses: globally, beeswarm and bar charts show which features matter most across thousands of predictions — useful for model debugging and feature selection. Locally, a waterfall chart for a single transaction shows exactly why it was flagged — essential when a customer calls to dispute a block, or when a compliance team needs an audit trail.
8
Why do you load the model at API startup rather than inside the endpoint function?
+
Model answer
Loading a scikit-learn model from disk takes around 200ms. At any meaningful request volume — say 100 requests per second — loading per-request adds 20 seconds of wasted I/O per second. The service would never keep up. Instead, we use FastAPI's lifespan context manager to load the model once when the server starts. Every request then reuses the same in-memory object at near-zero cost. The model is loaded exactly once; the threshold is read from config at the same time, so updating the threshold just means editing a YAML file and restarting the server — no retraining needed.
9
What is data drift and how would you detect it in this system?
+
Model answer
Data drift is when the statistical distribution of incoming features shifts away from what the model was trained on. Fraud patterns evolve constantly — new attack vectors, seasonality, changing merchant categories. A model trained six months ago may be silently worse today. To detect it I would: log every incoming request's feature values, compute daily feature distributions, and compare them to the training distribution using a statistical test like the Kolmogorov-Smirnov test or Population Stability Index. Tools like Evidently AI automate this. When drift is detected, it triggers a retraining job.
10
What would you add to this system if you had another week?
+
Model answer
Three things.

First, a retraining pipeline triggered by drift detection. The model should update automatically as fraud patterns change. When the KS test flags drift on a key velocity feature, it kicks off a training run on the latest labelled data, validates the new model against the current champion, and promotes it if recall improves.

Second, a graph neural network component. Fraudsters often operate in networks — the same device, IP, or card-not-present merchant appears across many victims. Individual transaction models miss ring fraud entirely. A GNN that models the transaction graph can detect those patterns. It would run alongside the current model, with the scores combined.

Third, a feedback loop dashboard so the fraud operations team can label flagged transactions directly. Those confirmed labels feed back into the next training run. The current system is a good first version — but production fraud systems need that closed loop to stay effective as attackers adapt.
