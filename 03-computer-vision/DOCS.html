What is this?

A production object detection service. You upload an image, and the API returns bounding boxes, class labels, and confidence scores for every object it finds — 80+ object categories from the COCO dataset.

The model is Faster R-CNN with a ResNet-50 FPN backbone — a two-stage detector that first proposes candidate regions, then classifies each one. More accurate than single-stage detectors like YOLO, with well-understood architecture decisions worth explaining in interviews.

Ships with a static HTML drag-and-drop frontend for live demos.

How it works — step by step

1
Upload — image sent as multipart form data to POST /detect
2
Decode — cv2.imdecode on raw bytes (faster than writing to temp file and opening with PIL)
3
Preprocess — BGR → RGB → transforms.ToTensor() normalises to float32 [0, 1]
4
Inference — torch.no_grad() forward pass through Faster R-CNN: backbone → RPN → RoI Align → classification head
5
Confidence filter — discard any box with score below threshold (default 0.5)
6
NMS — torchvision.ops.nms removes overlapping boxes for the same class (IoU threshold 0.2)
7
Return — JSON: bounding boxes [x1, y1, x2, y2], class names, scores, inference time

Faster R-CNN — two stages

Stage 1 — RPN:  ResNet-50 FPN backbone generates feature maps. Region Proposal Network slides over them and proposes ~2,000 candidate boxes with objectness scores.
↓
Stage 2 — Detection head:  For each proposal, RoI Align crops and resizes the feature map. A fully connected head predicts class probabilities (91 COCO classes) and bounding box refinements.

Both stages share the backbone — they run in a single forward pass.

uvicorn src.api.main:app --reload

Example detection call:

  -F "file=@photo.jpg" \

Interview Q&A

Click any question to read the answer.

▶ Explain Faster R-CNN. What are the two stages?

Stage 1 — Region Proposal Network (RPN): a small convolutional network slides over the ResNet-50 feature map and proposes ~2,000 candidate bounding boxes ("anchors") along with objectness scores (object vs. background).

Stage 2 — Detection head: for each proposal, RoI Align crops and resizes the feature map to a fixed size. A fully connected head then predicts class probabilities (91 COCO classes) and bounding box refinements. The classification and box regression run in parallel.

Both stages share the backbone — they run in a single forward pass. The "Faster" in the name refers to the fact that the RPN replaced the external selective search used in earlier R-CNN variants, making the system end-to-end trainable.

▶ What is Non-Maximum Suppression (NMS) and why is it needed?

NMS removes redundant overlapping detections of the same object. After the RPN generates thousands of proposals, many overlap significantly (multiple anchors fire on the same car, for example).

How it works: sort boxes by score → keep the highest-scoring box → remove any other box with IoU > threshold → repeat. IoU = area of overlap / area of union.

This project uses torchvision.ops.nms with threshold 0.2. Lower threshold = more aggressive suppression = cleaner results for the demo. In production this would be tuned per class.

▶ What is FPN (Feature Pyramid Network)?

FPN adds a top-down pathway to the ResNet backbone. ResNet produces feature maps at multiple scales (stride 4, 8, 16, 32). FPN merges these with lateral connections, producing multi-scale feature maps that combine:

  • High-resolution spatial detail from shallow layers (for detecting small objects)
  • Semantic richness from deep layers (for understanding what the object is)

Without FPN, a detector struggles to find both small objects (need high-res features) and large objects (need abstract features) in the same image. FPN solves this with minimal additional computation.

▶ Why Faster R-CNN over YOLO?

Faster R-CNN is a two-stage detector: more accurate, slower (~3–5s CPU). YOLO is single-stage: faster (~100ms), slightly less accurate at detecting small objects.

For a portfolio project demonstrating detection architecture, the two-stage design is more interesting to explain. The original codebase had a skeleton YOLODetector class (print statements only) — it was deleted. A production deployment prioritising real-time would use YOLOv8.

▶ What is the difference between confidence threshold and NMS threshold?

Confidence threshold — filters detections where the model is uncertain. Any box with score < threshold is discarded. Lower threshold = higher recall (fewer misses), more false positives.

NMS threshold — controls suppression of overlapping boxes for the same class. If two boxes predicting "person" overlap by more than the IoU threshold, the lower-scoring one is removed. Lower NMS threshold = more aggressive suppression = one box per object even in dense crowds.

▶ How would you evaluate this quantitatively?

Standard metric: mAP (mean Average Precision).

  • For each class, sweep the confidence threshold and plot the Precision-Recall curve.
  • AP = area under that curve for one class.
  • mAP = average AP across all classes.
  • COCO mAP@[0.5:0.95] averages across multiple IoU thresholds (0.5, 0.55, ..., 0.95) — more rigorous than single-threshold AP.

The pycocotools library handles this officially. For real-time systems also track FPS, p95 latency, and peak memory usage.

▶ How would you handle concurrent requests?

FastAPI is async but torch.no_grad() inference is synchronous (CPU-bound). Under concurrent load, requests queue behind each other's inference calls.

Solutions:

  • Run inference in a thread pool: await asyncio.get_event_loop().run_in_executor(None, detect_fn, image)
  • Batch requests: collect N images, run one batched forward pass
  • Multiple worker processes behind a load balancer (N × uvicorn workers or gunicorn)
▶ Why not ONNX for serving?

ONNX + ONNX Runtime would give ~2x faster CPU inference and simpler deployment (no PyTorch in production). For this portfolio demo, 3–5s latency is acceptable and PyTorch is simpler to set up.

Production pipeline: train in PyTorch → torch.onnx.export() → validate with onnxruntime → serve via ONNX Runtime or TensorRT.

Honest limitations

Inference is synchronous and CPU-bound (~3–5s per image). Blocks the event loop under concurrent load. Fix: run_in_executor.
COCO weights (~167MB) are downloaded on first startup. Should be pre-downloaded in Dockerfile for faster cold starts.
NMS is applied globally. Per-class NMS gives better results for overlapping instances of different classes.
No batching — processes one image per request.
The visualization.py module (MetricsVisualizer, VideoVisualizer) is not wired to any API endpoint — available as a utility but not exposed.
