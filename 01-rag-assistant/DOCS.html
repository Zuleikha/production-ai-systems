What is this?

A Retrieval-Augmented Generation (RAG) system. You give it documents — PDFs, text files, markdown — and you can ask questions in plain English and get grounded answers with inline citations.

The key idea: the LLM only answers from your documents, not from its training data. Every claim is cited back to the source chunk. This eliminates hallucination for in-domain questions.

The interesting part is the retrieval architecture — it combines dense vector search with BM25 keyword search, fuses the results, then reranks with the Cohere API before passing context to the LLM. That two-stage pattern is what separates production RAG from demo RAG.

How it works

Ingestion (once per document)

1
Load — PDF text extracted page by page with pypdf. Text files read directly.
2
Chunk — Split at sentence boundaries using tiktoken token counting. 512 tokens per chunk, 64-token overlap.
3
Embed — Each chunk sent to OpenAI text-embedding-3-small → 1536-dimensional vector.
4
Store — Chunk text + embedding + metadata upserted into Pinecone serverless index.

Query (every user question)

1
Embed query — Same text-embedding-3-small model.
2
Hybrid retrieve — Dense (Pinecone ANN) + Sparse (BM25) run in parallel → top 20 each.
3
RRF fusion — Reciprocal Rank Fusion combines the two lists. No weights to tune.
4
Rerank — Top 20 sent to Cohere Rerank API. Scores each (query, chunk) pair jointly → top 3 survive.
5
Generate — Top 3 chunks become numbered context. GPT-4o-mini instructed to cite by number [1] [2].
6
Return — Answer, cited sources, token count, per-stage latency.

API → localhost:8000  ·  Swagger → localhost:8000/docs  ·  Frontend → localhost:8501

Ingest a document:

curl -X POST localhost:8000/ingest -F "file=@report.pdf"

Ask a question:

curl -X POST localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"question": "What were the key findings?", "conversation_id": "s1"}'

Interview Q&A

Click any question to read the answer.

▶ What is RAG and why does it matter?

RAG grounds LLM answers in real documents rather than training data. Without RAG, an LLM answers from memory — which may be outdated or wrong for your specific domain. With RAG, the model only sees your documents at inference time. Every claim is verifiable. This eliminates hallucination for in-domain questions and makes the system auditable.

▶ What is the difference between a bi-encoder and a cross-encoder?

A bi-encoder encodes the query and each document separately into the same vector space. Retrieval is a nearest-neighbour lookup — fast and scalable, but the two texts never interact during encoding.

A cross-encoder takes the concatenated pair [query, document] as a single input. Every attention head can look at both query and document tokens together. This produces much more accurate relevance scores — but it requires one forward pass per candidate, so it's too slow for large corpora.

This system uses bi-encoder for retrieval (fast, top 20 candidates) and Cohere's cross-encoder reranker for scoring (accurate, small candidate set). Best of both worlds.

▶ Why hybrid retrieval — dense + BM25?

Dense search finds semantic matches — "what causes high interest rates?" finds text about monetary policy even without those exact words. But it can miss specific named entities: model numbers, person names, rare technical terms.

BM25 does exact keyword matching — perfect at finding "Sarah Mitchell" or "CFO Q3" but blind to paraphrasing.

Each covers the other's failure mode. The cost is negligible — BM25 adds ~10ms. RRF fusion combines both lists without any tuning parameters.

▶ What is RRF and why use it over weighted scoring?

RRF formula: score = sum(1 / (k + rank_i)) where k=60.

A weighted combination like 0.7 × dense + 0.3 × bm25 requires tuning those weights, and the right weights are domain-specific — they differ for legal documents vs technical manuals. You'd need labelled data.

RRF only uses rank positions, not raw scores — so it doesn't matter that cosine similarity and BM25 scores have different scales. It's parameter-free and empirically robust across domains since the original 2009 paper.

▶ Why Cohere Rerank API instead of a local model?

A local cross-encoder (e.g. cross-encoder/ms-marco-MiniLM-L-6-v2 via sentence-transformers) requires PyTorch. Even the CPU-only build allocates 300–400 MB of RAM at import time. On Render free tier (512 MB limit), this crashes the service before the first request is served.

The Cohere Rerank API is a single HTTP POST via httpx — zero local model, zero PyTorch, zero RAM overhead. Equivalent cross-encoder quality server-side. Free tier covers 1,000 requests/month. If the key is absent or the call fails, the system falls back to embedding-score ordering gracefully.

▶ Why Pinecone instead of ChromaDB?

ChromaDB runs in-process and persists to local disk. Two problems for production:

  • Ephemeral storage — Render free tier has no persistent disk. Every redeploy wipes the index.
  • RAM — ChromaDB's HNSW index loads into process memory alongside the app.

Pinecone is a managed serverless vector database. The index persists in Pinecone's infrastructure, survives redeploys, and has zero in-process RAM cost. The tradeoff is an API call per query (~50–100ms latency), which is acceptable.

▶ What are the failure modes?
  • Out-of-distribution questions — if the answer isn't in the documents, the LLM may still stretch the retrieved context. Hallucination at the boundary is the hardest to catch.
  • Large document corpora — BM25 rebuilds from scratch on every query (no caching). For 10k+ chunks this becomes slow.
  • Vague queries — short ambiguous questions produce weak embeddings and poor BM25 signals.
  • Chunking artifacts — the sentence boundary regex fails on bullet points, tables, and code blocks.
  • Cohere API dependency — if the API is down, reranking falls back to embedding-score ordering. Quality degrades but the service stays up.
▶ How would you evaluate retrieval quality?

Retrieval metrics:

  • Recall@k — of all chunks containing the answer, what fraction are in the top-k?
  • MRR — Mean Reciprocal Rank. 1 / rank_of_first_relevant_chunk.
  • NDCG — accounts for graded relevance and position.

End-to-end quality (RAGAS framework):

  • Faithfulness — is every claim grounded in the retrieved context?
  • Answer relevance — does the answer address the question?
  • Context precision — of retrieved chunks, how many were actually used?
  • Context recall — does the context contain all needed information?
▶ How would you scale this?
  • Cache BM25 index — currently rebuilt on every query. Store in Redis, update on ingest.
  • Async Cohere call — switch from synchronous httpx.post to httpx.AsyncClient to avoid blocking the event loop under concurrent load.
  • Query cache — LRU cache on the question string so repeated queries skip OpenAI calls.
  • Pinecone namespaces — isolate document sets per user or organisation.

▶ Why not use LangChain or LlamaIndex?

I know both well. The decision was deliberate. For a portfolio project, native implementation makes the architecture visible. When someone reads HybridRetriever.retrieve(), they see the BM25 call, the dense call, and the RRF fusion in plain code.

LangChain was removed specifically because it was only used for RecursiveCharacterTextSplitter — an 80-line replacement. That's a bad dependency ratio.

In production I'd reconsider — LangChain's integration ecosystem (dozens of vector stores, tracing callbacks) has real value. LlamaIndex is particularly good for complex document structures. The tradeoff is abstraction vs. transparency.

Honest limitations

BM25 index rebuilds from scratch on every query. For 10k+ chunks this would be noticeably slow. Production fix: cache in Redis, update on ingest.
No evaluation harness. RAGAS would be added before treating this as production-ready.
Conversation history is in-process memory. Restarting the API loses all conversations. Production fix: persist to Redis with a TTL.
The sentence boundary regex fails on bullet points, tables, and code blocks.
No authentication on API endpoints. Any caller can ingest or query.
The Cohere rerank call is synchronous inside the async pipeline. Under concurrent load this blocks the event loop. Fix: switch to httpx.AsyncClient.
Pinecone free tier has index size limits — fine for a portfolio demo, not for large corpora.
