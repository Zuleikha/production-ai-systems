What is this?

An AI-powered resume screening system. You give it a job description and a set of resumes (PDF, DOCX, or plain text), and it ranks candidates by how semantically similar their experience is to the role requirements.

Unlike keyword matching, semantic similarity understands that "built microservices" and "developed distributed services" mean the same thing. It uses a sentence-transformer model to encode both the job description and each resume as a vector, then scores them by cosine similarity.

It also includes a Pydantic v2 validation layer (schema compliance, date checking, duplicate detection), a data quality reporter, and a bias monitor for demographic parity analysis — important components for responsible deployment of automated hiring tools.

How it works — step by step

1
Parse — PDF via pypdf, DOCX via python-docx, plain text directly.
2
Validate — each resume checked against ResumeSchema (Pydantic v2): required fields, email regex, date format, cross-field date logic.
3
Embed — all-MiniLM-L6-v2 from sentence-transformers encodes resume text → 384-dimensional vector.
4
Score — cosine similarity between each resume vector and the job description vector.
5
Rank — candidates sorted by score, assigned a tier: strong / good / borderline.
6
Report — data quality metrics + bias analysis (demographic parity, equal opportunity).

Data schema — what gets validated

Every resume is validated against ResumeSchema (Pydantic v2) before it reaches the embedding pipeline:

ContactInfo — email (regex-validated), optional phone, LinkedIn, GitHub
Skill — name, optional level (beginner / intermediate / advanced / expert), years of experience
Education — degree, institution, graduation year, GPA
Experience — title, company, ISO date range, description (20–2000 chars), skills used
Cross-field validator — checks start_date < end_date for every experience entry. Any violation raises a ValidationError with field-level detail before the record reaches the model.
Bias note: automated resume screening is a high-stakes application. The BiasMonitor measures demographic parity (equal selection rate across groups) and equal opportunity (equal true positive rate). Its presence is a deliberate design decision — not just a feature checkbox.

Interview Q&A

Click any question to read the answer.

▶ How does semantic similarity between a resume and job description actually work?

Both the resume text and job description text are passed through the same sentence-transformer model, producing fixed-length vectors (384 dims for MiniLM).

Cosine similarity = dot_product(A, B) / (|A| × |B|) → score in [-1, 1] where 1 = identical direction.

Unlike keyword matching, semantic similarity captures paraphrasing: "built microservices" ≈ "developed distributed services". Candidates with scores closer to 1.0 are semantically more aligned with the job description.

▶ Why cosine similarity instead of euclidean distance?

Embeddings encode direction, not magnitude. A long resume and a short resume about the same candidate would have different vector lengths, making euclidean distance misleading — the longer document would appear "farther away" just because it's longer.

Cosine similarity measures the angle between vectors — it's length-invariant. This is the standard for comparing sentence embeddings and is what the models were trained to optimise.

▶ What is all-MiniLM-L6-v2 and why this model?

all-MiniLM-L6-v2 is a 22M parameter model fine-tuned on 1 billion+ sentence pairs for semantic similarity (SBERT: Sentence-BERT). It produces 384-dimensional embeddings.

Why this over a larger model? It's fast (suitable for batch processing many resumes), small enough to run on CPU, and strong enough for resume-job matching. The larger alternative (all-mpnet-base-v2) has 110M parameters and 768 dims — ~5x slower with modestly better accuracy. For initial screening at volume, MiniLM is the right tradeoff.

▶ Walk me through the Pydantic ResumeSchema. What does the cross-field validator check?

@field_validator("skills") enforces max 50 skills per candidate.

@field_validator("experience") iterates all experience entries and checks that start_date <= end_date (when end_date is not "present") using ISO date string comparison — "2020-01-01" > "2019-06-01" works lexicographically.

ContactInfo applies regex patterns directly in Field(pattern=...) — Pydantic v2 evaluates these at the class level using the annotated-types Rust core. Any validation failure raises ValidationError with field-level detail before the record reaches the embedding pipeline.

▶ What bias can occur in embedding-based resume screening?

Several types:

  • Training data bias — the sentence-transformer was trained on web text, which reflects societal biases. It may score resumes from underrepresented groups lower even for equivalent experience.
  • Proxy discrimination — similarity to "ideal candidate" descriptions can encode gender/age/ethnicity via correlated language patterns.
  • Recency bias — candidates from highly-ranked institutions may use the same language as the job description authors from those institutions.

The BiasMonitor measures demographic parity (equal selection rate across groups) and equal opportunity (equal true positive rate across groups). Real bias monitoring requires labelled demographic data — the current implementation is a structured placeholder that demonstrates the design intent.

▶ How would you scale from 100 to 100,000 resumes?
  • Batch embeddings — model.encode(texts, batch_size=64) instead of one-at-a-time. sentence-transformers already supports this.
  • Vector store — rather than computing similarity from scratch each query, store embeddings in a vector database (Pinecone, pgvector, ChromaDB) and use ANN search. Current design is O(N) per query.
  • Async processing — move parsing + embedding to a task queue (Celery, Redis Queue) so the API returns immediately and results are polled.

Under 1,000 resumes, the current design is fine. Above that, the ANN search becomes the bottleneck.

▶ Why did you remove SQLAlchemy from requirements?

It was listed in requirements.txt but not imported anywhere in src/. This is a common inherited dependency smell — someone planned a database layer, added the requirement, but never implemented it.

Keeping it adds install time, potential security surface area, and confusion for code reviewers. If a database layer is added later, the requirement can be restored with intention.

▶ How would you handle scanned PDFs (images, not digital text)?

pypdf extracts digital text — it returns empty strings for scanned PDFs (images of text).

Solution: OCR fallback. If extract_text() returns empty or near-empty text, convert the PDF pages to images and run pytesseract (wraps Tesseract OCR) or a cloud service (AWS Textract, Google Document AI). The PDFParser would need a fallback branch — a known limitation of the current implementation.

▶ What does DataQualityMetrics.calculate_quality_score compute?

quality_score = (valid_records / total_records) × 100

The simplest possible quality metric — the percentage of records that passed all Pydantic schema validations. A score of 85% means 15% of uploaded resumes had validation errors (missing fields, malformed dates, invalid emails).

A more sophisticated metric would weight fields by importance — an invalid email is more critical than a missing LinkedIn URL.

Honest limitations

No vector store — similarity is O(N) per query. Above ~1,000 resumes, ANN search would be needed.
BiasMonitor returns hardcoded placeholder values — real bias monitoring requires labelled demographic data.
DataQualityChecker.validate_schema_compliance calls .dict() (Pydantic v1 method) — should be .model_dump() in Pydantic v2.
No OCR fallback — scanned PDFs return empty text and fail silently.
feature_engineering.py and preprocess_data.py at the project root are standalone scripts that don't integrate with the src/ module structure.
The FastAPI main.py is minimal — src/api/schemas.py defines models but the running API is not fully wired end-to-end in the current state.
