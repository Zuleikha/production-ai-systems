What is this?

An end-to-end MLOps system that takes a raw dataset all the way to a deployed model serving API. The whole pipeline is orchestrated as Dagster assets — each stage is a named data artefact with declared dependencies.

Specifically: it fine-tunes BERT on the IMDB sentiment dataset, tracks every experiment in MLflow, and serves predictions through a FastAPI endpoint. You can trigger a full retrain via the API, or re-run any single stage from the Dagster UI without touching the others.

The point of this project is to demonstrate the full MLOps lifecycle — not just training, but orchestration, experiment tracking, model registry, and deployment as a connected system.

Pipeline assets — 9 stages in order

1
pipeline_config — loads and validates config/pipeline.yaml
2
dataset_info — resolves dataset name, split sizes, label names from HuggingFace
3
raw_dataset — downloads IMDB (25k train, 25k test) from HuggingFace Hub
4
validated_dataset — pandas assertions: no nulls, correct columns, text length bounds
5
preprocessed_dataset — tokenises with bert-base-uncased, encodes labels
6
trained_model — fine-tunes BERT with HuggingFace Trainer, logs to MLflow
7
evaluated_model — accuracy, F1, confusion matrix on the test set
8
model_registry — saves artefacts and metadata to MLflow model registry
9
deployment_config — writes serving config for the FastAPI layer

You can re-materialise any single asset (e.g. just trained_model) without re-running the full DAG. Dagster handles dependency ordering automatically.

Quick start

pip install -r requirements.txt
cp .env.example .env   # set MLFLOW_TRACKING_URI=./mlruns

Start Dagster UI:

dagster dev -f src/dagster_definitions.py
# open http://localhost:3000 → click Materialize All

View MLflow experiments:

mlflow ui
# open http://localhost:5000

Start serving API:

uvicorn src.api:app --reload
# open http://localhost:8000/docs

Interview Q&A

Click any question to read the answer.

▶ What is the Dagster asset model and why is it better for ML than Airflow?

In Dagster, you declare what data artefacts exist and how to produce them — not which tasks run in which order. Each @asset is a named artefact (e.g. trained_model) with declared upstream dependencies. The scheduler materialises them in topological order.

Benefits:

  • Partial re-runs — re-materialise just trained_model without re-downloading the dataset.
  • Built-in lineage — Dagster knows which data produced which model.
  • Independently testable — each asset can be unit-tested without the full pipeline.

Airflow defines task execution order — there's no concept of "the trained model" as a first-class object. Re-runs and partial backfills are much harder.

▶ Why MLflow instead of Weights & Biases?

MLflow is fully self-hosted — no external service, no API key, no data leaving the machine. For a portfolio project it eliminates setup friction. MLflow handles parameter logging, metric curves, artefact storage, and a model registry. The original code used both wandb AND mlflow redundantly — that was removed.

W&B is excellent in team environments where you want shared dashboards and collaboration. MLflow is better when you need full local control.

▶ Why use HuggingFace Trainer instead of a custom training loop?

Trainer handles gradient accumulation, mixed-precision, evaluation callbacks, checkpoint saving, and best-model selection — all with battle-tested defaults. A custom loop duplicates this with worse defaults and no community maintenance.

Use a custom loop when you need unusual loss functions, custom schedulers, or multiple optimisers (e.g. GANs). For fine-tuning a standard BERT model, Trainer is the right tool.

This project kept a custom PyTorch loop in _train_custom_model to demonstrate the pattern — but the primary path uses Trainer.

▶ How does MLflow experiment tracking work here?

Each mlflow.start_run() opens a run context. The pipeline logs:

  • Parameters — model name, batch size, learning rate, epochs via mlflow.log_params
  • Metrics — train/val loss per epoch via mlflow.log_metrics(step=epoch)
  • Artefacts — model weights, tokenizer config via mlflow.log_artifacts

The MLflow UI at localhost:5000 shows all runs, lets you compare loss curves, and download artefacts. The model registry versions and stages models (staging vs. production).

▶ How would you scale this to 10x more data?
  • Data — use streaming=True in load_dataset to avoid loading everything into RAM. Process in chunks.
  • Training — enable fp16=True in TrainingArguments for ~2x throughput. Use gradient accumulation to simulate larger batches without more memory.
  • Orchestration — Dagster supports partitioned assets. Partition by date or data shard and materialise partitions in parallel.
▶ Is the /train endpoint safe for concurrent requests?

No — it's a race condition. If two /train requests run concurrently, they will both try to write pipeline.yaml at the same time, corrupting the config.

In production, you'd queue training jobs (via Dagster's run queue or a message queue like Redis Queue) rather than triggering them synchronously via HTTP. The direct mutation is a deliberate simplification for local demo purposes.

▶ What monitoring would you add in production?

Three layers:

  • Infrastructure — Prometheus scraping request latency, error rate, and throughput. Already wired with prometheus-client.
  • Data drift — Evidently monitoring input text distributions vs. the training distribution. Alert when drift exceeds a threshold.
  • Model performance — periodically sample predictions, label them, recompute accuracy/F1. Trigger retraining if metrics degrade below a threshold.
▶ How would you add A/B testing for two model versions?

Route traffic at the API layer: maintain two ModelService instances (v1 and v2), randomly assign each request based on a configured split (e.g. 90/10), log which version served each request, then compare metrics in MLflow or Grafana.

MLflow's model registry supports staging (staging vs. production) — promote v2 to production when its metrics exceed v1's. This maps cleanly to the A/B testing workflow.

Honest limitations

The /train endpoint is synchronous and not idempotent — concurrent calls will corrupt pipeline.yaml. Production fix: use a job queue.
Model checkpoint paths are relative — the API will fail to load if started from a different working directory.
Grid search is exhaustive — for large hyperparameter grids, Bayesian optimisation (Optuna) would be significantly faster.
The DataManager class in data_processor.py is partially unused — designed for a custom model path that was later superseded by the Dagster asset pipeline.
Two separate API files (src/api.py and src/deployment/api.py) — in a real project these would be merged.
